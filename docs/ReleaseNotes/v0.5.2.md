# M5 Cardputer Wardriver — v0.5.2 Release Notes

**Release date:** April 21, 2026
**Target hardware:** M5Stack Cardputer v1.1 and Cardputer ADV (ESP32-S3)
**Firmware binary:** `output/wardriver-0.5.2.bin`

---

## Highlights

- Added **Cardputer ADV support** through a new hardware-profile system
- A single firmware binary now supports both **Cardputer v1.1** and **Cardputer ADV**
- Firmware now **auto-detects the board model** at boot and selects the matching hardware profile
- New **Device** section in the web portal for hardware model selection and troubleshooting
- Startup splash and **Dashboard F** now show the active hardware model and whether it came from **auto** detection or a **manual** override
- Existing configs upgrade safely: older installs gain model tracking without resetting custom GPS pin overrides on first boot

---

## New Features

### Cardputer ADV Hardware Profiles

Firmware v0.5.2 introduces a dedicated hardware-profile layer so the wardriver can adapt to multiple Cardputer variants at runtime.

The new profile system currently supports:

| Model | Profile key | Default GPS TX | Default GPS RX |
|-------|-------------|----------------|----------------|
| Cardputer (v1.1) | `cardputer` | `2` | `1` |
| Cardputer ADV | `cardputer_adv` | `13` | `15` |

At boot, the firmware queries `M5.getBoard()` and maps the result to the correct profile automatically. If detection is unavailable, the firmware falls back to the original Cardputer profile so the device still boots predictably.

### Web Portal Device Section

The configuration portal now includes a new **Device** section at the top of the page.

It provides:

- A hardware model selector with `Auto-detect`, `Cardputer (v1.1)`, and `Cardputer ADV`
- A read-only **Detected** value showing what the firmware found at runtime
- A read-only **Active** value showing the effective profile currently in use
- A firmware version display for quick support checks
- Default GPS pin hints for both supported models

When the selected hardware model changes, the portal reseeds profile-driven defaults such as GPS TX/RX to match the chosen board profile, then saves and reboots.

### On-Device Model Visibility

To make troubleshooting easier, the firmware now exposes the active hardware profile directly on the device UI:

- The startup splash shows `Model: <name> (<source>)`
- Dashboard F now shows the active **Model** and **Src** values during normal operation
- Serial startup logs print the detected board model and the effective profile selected for this boot

This makes it much easier to confirm whether the device is running in auto-detected mode or under a manual override.

## Compatibility And Migration

v0.5.2 is designed to preserve existing behaviour on Cardputer v1.1 units while adding multi-board support.

### Existing installs

- Upgrading from an earlier `0.5.x` release keeps existing settings intact
- Older configs gain a persisted `detected_model` field on first boot after upgrade
- If an older config has custom GPS TX/RX pins, those values are not overwritten just by upgrading

### SD card moved between boards

If you move the same SD card between a Cardputer v1.1 and a Cardputer ADV, the firmware compares the stored model state with the newly detected board and can reseed profile-driven defaults automatically.

This helps the device adapt to a board change without forcing a full config reset.

## Configuration Changes

Two new device-level settings are now persisted in `/wardriver/wardriverconfig.txt`:

| Key | Default | Description |
|-----|---------|-------------|
| `device.model` | `"auto"` | User-selected hardware mode: `auto`, `cardputer`, or `cardputer_adv` |
| `device.detected_model` | `""` | Runtime-populated record of the effective hardware model used for migration and change detection |

Example snippet:

```json
{
  "device": {
    "model": "auto",
    "detected_model": "cardputer"
  }
}
```

## Changes

### Internal Architecture

This release adds a new `HardwareProfile` module that centralizes board-specific defaults and pin assignments.

This is the main foundation for future hardware support work and avoids maintaining separate firmware builds for each Cardputer variant.

### UI And Supportability

- The startup splash now includes the detected or selected hardware model
- Dashboard F now includes hardware model metadata alongside the existing system information
- The web portal surfaces the effective profile directly instead of leaving model selection implicit
- Serial/debug output now includes model-detection context to simplify field troubleshooting

### Behaviour Preserved

Core wardriving behaviour from v0.5.1 is otherwise unchanged:

- Six dashboard views remain available
- `SPACE` still cycles dashboard sub-modes where supported
- WiGLE CSV logging, GPS modes, filters, and scan controls behave as before

---

## Technical Details

- Board detection is performed via `M5.getBoard()` after `M5Cardputer.begin()`
- Unknown or unsupported detections fall back to the Cardputer v1.1 profile
- Profile-driven reseeding is limited to hardware-dependent defaults so unrelated settings such as scan timing, brightness, and buzzer frequency are preserved
- The web portal only triggers reseeding when the submitted form includes a hardware model change, which avoids clobbering manual GPS pin overrides during unrelated saves
